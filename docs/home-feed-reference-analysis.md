# 홈/메인 노출 분류 기준 — 레퍼런스 3사 분석 (2026-06-02)

> 홈 신선도·날짜 모델 설계 중 "메인 화면 콘텐츠를 어떤 기준으로 구획하나"를 뉴닉·요즘IT·Toss Feed로 조사. [[project-web-calendar-banner]]·발행/신선도 모델 결정의 근거.

## 3사 비교

| 차원 | 뉴닉 | 요즘IT | Toss Feed |
|---|---|---|---|
| 분류 축 | 시리즈(요일별 레터)+시사 카테고리+"이슈+"(주제별 총정리) | 카테고리(개발·AI…)+편집 컬렉션 | 관심주제+필진+시리즈 |

## 가로지르는 핵심 패턴

1. **누구도 홈을 "날짜"로 구획하지 않는다.** 조직 원리 = 에디터픽 + 인기 + 신선도(한 레인) + 주제/관심사 + 필진/시리즈 + 카테고리.
2. **신선도는 '한 섹션'이지 '척추'가 아니다.** "방금 올라온"/"새로 나온"은 여러 섹션 중 하나.
3. **카드에 발행일을 안 보여준다.** 대신 읽기시간 + 카테고리 태그 + NEW 배지로 신선도·맥락.

## ★ 우리에게 적용 (publisher vs aggregator 구분)

- **레퍼런스 3사 = 1차 저작(publisher)** → 발행일=자기가 쓴 날 → "새로 나온" 섹션이 곧 신선도. 카드에 날짜 불필요.
- **우리 = 외부 뉴스 큐레이터(aggregator)** → 남의 콘텐츠(교육정책·박물관행사)를 긁어 보여줌 → **원문 발행일이 정보값+신뢰 장치**. 그래서 날짜 표시 결정은 레퍼런스와 정반대로 감(이건 모델 차이).
- **가져온 것**: "섹션을 날짜로 구획하지 마라" → 홈 ① 헤더 "{날짜}의 큐레이션" → **"새로 올라온 소식"**.
- **안 가져온 것**: "카드에서 날짜 빼라"(그들은 publisher라서) → 우리는 카드에 **원문 발행일** 유지.

## 채택 설계 (Phase A, 마이그레이션 0 — 구현·검증 완료 2026-06-02)

- 카드 날짜: `issueDate` → **`publishedAt`(원문 발행일)**.
- 홈 ① 섹션 헤더: 날짜 → **"새로 올라온 소식"**. 가짜 🆕 제거.
- 정렬·신선도 = `issueDate`(=우리가 가져온 날, 큐레이션이 `kstIssueDate(실행일)`로 찍음) → 늦게 가져온 속보도 그날 묶음 상단 = 묻힘 해결.
- 날짜 기반 정리는 `/issues` 아카이브에 보존.
- **남은 여지(프로토타입 보고 개선)**: ⓐ 신선도를 issueDate(일 단위)보다 정밀하게(intra-day) = revealedAt 전용 컬럼(Phase B, 멀티발행 파이프라인 생길 때) ⓑ 토스식 섹션 다양화(인기·주제·필진 — toss-feed-analysis.md 패턴) ⓒ 읽기시간 표시 검토.
