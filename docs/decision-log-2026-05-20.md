# 페런트웹 SPEC 의사결정 학습 노트 (2026-05-20)

기획 SPEC.md를 정리하며 내린 설계 결정 A~F의 사고 과정. 결정 자체보다 **반복해서 쓴 판단 메커니즘**에 주목 — 이건 다음 결정에도 그대로 전이된다.

> 역할 분리: [SPEC.md](SPEC.md)는 "현재/목표 상태(무엇을 어떻게)", 이 로그는 "왜 그렇게 됐나(판단 근거)".

---

## Part 1. 전이되는 판단 메커니즘 (메타 교훈)

### 1. 결정값 vs 파생값을 분리하라
발행 카드 수(제품 결정값) → 리서치 수집량(파생값). 손잡이는 하나여야 한다. 무엇이 원인이고 무엇이 결과인지 먼저 정한다.

### 2. 진짜 불변값을 찾아 거기에 고정하라
지키려는 건 절대 숫자가 아니라 **압축률(엄선 강도)**. 그래서 `SELECTION_FACTOR(×2)`를 불변으로 두고 나머지를 파생. 불변값을 잘못 잡으면 엉뚱한 걸 고정한다.

### 3. 실제 업계/직군에 매핑해 검증하라
에이전트 분업을 실제 뉴스룸 직군(스카우트/데스크/기자/카피에디터/소셜PD)에 대보니 "취향"이 아니라 검증된 분업 구조임이 드러남. "현실에서 이 일을 하는 사람은 누구이고 어떻게 나뉘나?"를 먼저 본다.

### 4. greenfield인지 refactor인지로 "지금 vs 나중" 비용을 판단하라
"일단 합치고 나중에 분리" 격언은 **돌아가는 엉킨 코드를 리팩터할 때** 참. 아직 안 만든 상태(greenfield)면 분리 비용이 낮다. 격언을 상황에 맞게 뒤집을 줄 알아야 한다.

### 5. 비용의 비대칭을 보라 (언제·몇 번 내는가)
미사용 필드 유지 = 캐리 비용을 **매번·계속·무조건** 지불. 삭제 후 재추가 = **한 번·필요할 때만**. 미래가 불확실하고 재추가가 싸면 삭제가 유리. "유지 vs 삭제"가 아니라 **비용 타이밍**으로 본다.

### 6. YAGNI는 "투기적 기능"에만 — "확정 로드맵"과 구분하라
반드시 할 게 확정이면 미리 둔다. 불확실하면 필요할 때 만든다. 핵심 질문: "확정된 지연인가, 막연한 가능성인가?"

### 7. 단일 책임이 LLM 프롬프트 품질을 올린다
"골라+쓰고+다듬어라" 한 프롬프트 = 작업이 평준화. 핵심 가치(본문 보이스)일수록 전담 에이전트로 떼어 독립 튜닝. 에이전트를 늘리는 정당한 이유 = **비용을 더 내도 품질이 그만큼 오를 때**.

### 8. substrate(토대) 먼저, layer(겉) 나중
writer는 정확성·구조·정책(S1~S9)이 토대, 브랜드 보이스는 그 위 레이어. 토대 없이 겉만 얹으면 "예쁘게 틀린 글". 순서를 지킨다.

### 9. 제약이 아키텍처를 강제한다
"전문 저장 금지"(저작권) → writer는 WebFetch로 transient read만, body는 transformative. 정책·법적 제약을 먼저 이해하면 구조가 따라 나온다.

### 10. 의도는 죽은 코드가 아니라 문서로 증언하라
미사용 컬럼을 남기는 이유가 "로드맵 표현"이면 SPEC 노트가 대신할 수 있다. 캐리 비용 없이 방향성만 보존, 코드는 깨끗하게.

### 11. 가역성을 확보하면 결정이 가벼워진다
발행 수·배율을 config 한 곳에 두면 "soft, 언제든 재조정". "되돌리기 비싼가?"를 먼저 물으면 싼 결정은 빨리 내릴 수 있다.

### 12. 실패는 "의존성 종류"로 분기하라
모든 실패를 똑같이 다루지 않는다. 뒷단계가 **반드시 필요로 하는** 산출(후보·선별)이 없으면 **중단**, 없어도 degrade로 발행 가능한 산출(본문·교열·후킹)이면 **계속**. 실패 정책 = 데이터 의존 구조에서 나온다.

## Part 2. 결정별 Q&A 로그

### A. 하루 노출 카드 수 (8~20, 배율 ×2)
- **혼란**: "카테고리 7개인데 카드 5~7개?" → 카테고리(라벨) ≠ 카드 수(노출량). 직교한다.
- **재구성**: 발행 풀(curation 산출) vs 유저가 보는 양(V2 세그먼트 필터)이 분리됨. 세그먼트를 지탱하려면 풀이 두툼해야 → 5~7은 부족.
- **배율**: 압축률을 불변(×2)으로. 리서치는 단일 상한(~40) best-effort, 발행 8~20은 비율 축소가 아니라 **품질 컷**.
- **결론**: publish 8~20, factor ×2 → research ~40. 부실한 날(통과분<8)은 실패 아님(품질>수량). 전부 config·가역적.

### B. Article 행을 누가 만드나 (curation = B1)
- **"DB 행을 만든다"의 뜻**: 후보 JSON 덩어리(AgentLog.output) → 표의 정식 한 줄(Article). 그래야 id·검색·상세페이지·카드 노출 가능.
- **결론**: curation이 선별(발제 승인) 시 draft 행 생성 = 실제 CMS 워크플로(데스크 승인=draft 생성)와 일치. `(issueDate,url)` unique로 중복 방지.

### C. 카드뉴스 body를 누가 쓰나 (writer 분리 = C2)
- **발견**: CLAUDE.md가 강조하는 body를 어느 에이전트도 출력 안 함(공백).
- **직군 매핑**: 고르는 자(데스크/curation) ≠ 쓰는 자(기자/writer) ≠ 다듬는 자(카피에디터/editor).
- **지금 분리?**: greenfield라 비용 낮음 + 본문 보이스가 핵심 차별점 + 작업 단위 다름(집필=건당/교열=이슈당) → **지금 분리**. writer 신설(4→5 에이전트).

### Writer 필수 스킬 (substrate)
- S1 소스독해 / S2 사실그라운딩 / S2′ transformative / S3 양육자 시사점 / S4 카드뉴스 구조화 / S5 가독성 / S6 타입별 작법 / S7 간결성 / S8 출처·링크 / S9 정책준수.
- **핵심 발견**: 150자 요약만으론 얕은 글 → writer는 `WebFetch`로 소스를 실제로 읽어야(단일 url만). 브랜드 보이스는 이 위 레이어.

### 저작권 — 전문을 저장하지 않는 이유
- 기사 전문 저장·게시 = 복제권·공중송신권 침해. 링크는 비침해, 짧은 인용은 정당 인용.
- 우리 모델(링크+자체 요약+자체 본문)은 합법 구간(transformative+link). → writer는 읽되 저장 안 함, body는 우리 말로(near-verbatim 금지).
- 부차 이유: 신선도·저장 위생·"호스트 아닌 큐레이션 레이어" 포지셔닝.

### D. hooking 매핑 + 미사용 필드 (단순화 = YAGNI)
- **매핑**: title←cardHook, summary←homeCopy, theme←editor.
- **유지 vs 삭제**: 미래 불확실 + 재추가 저렴 + 캐리비용 계속 + 제거 시 드리프트(Prisma↔shared) 해소 → **삭제**.
- **결론**: summary 컬럼만 추가, 이메일 필드(subjectLine/previewText/hookingCopy)·SNS 출력(visualTheme/seoHeadline) 제거. 의도는 SPEC deferred 노트로 보존, 기능 착수 시 nullable 재추가.
- **이메일 필드 뜻**: subjectLine=받은편지함 제목, previewText=프리헤더, hookingCopy=메일 본문 첫 훅 (이메일 전용 → 웹 MVP 미사용).
- **visualTheme 뜻**: SNS 공유 카드/썸네일 디자인 지시서(배색·이모지·이미지타입). 이미지 생성은 V1.5/V2.

### E. 파이프라인 오케스트레이터 (runPipeline 구현)
- `runPipeline(issueDate)`: research→curation→writer(건당 fan-out)→editor→hooking. 각 단계는 직전 AgentLog(success).output을 입력으로.
- **실패 = 의존성 분기**(메커니즘 12): research·curation 실패 → 중단 / writer·editor·hooking 실패 → degrade 후 계속.
- 멱등(issueDate upsert + url unique) → 재실행 안전. dry-run = 발행성 쓰기 미커밋(미리보기·프롬프트 검증).

### F. AgentConfig (DB 프롬프트 버전관리) — V2 보류
- 프롬프트는 파일(prompt.md) + git 버전관리로 충분(하드코딩 금지 원칙 충족).
- 테이블은 보존하되 미사용 — 운영 중 무중단 프롬프트 교체가 실제 필요해지는 V2에 활성화. (이메일 필드와 동일한 "보존하되 비활성" 패턴, 메커니즘 6·10)
