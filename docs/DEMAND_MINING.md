# 수요 발굴 (Demand Mining) — 부모 검색 수요로 큐레이션·유통을 정렬

> **목적**: 부모가 *실제로 검색·고민하는 것*을 데이터로 캐서 ① 큐레이션·발행 토픽 우선순위(**공명**) ② SEO 랜딩·시즌 캘린더(**유통**)에 반영. CSO 진단(비타민→진통제, sub-PMF)의 *공명+유통 병목*을 직접 겨냥.
> **척추 = 시즌성**. 진통제 쐐기(취학 전후 신청·마감)가 본질적으로 시즌성이고, 이미 만든 [웹 키데이트 캘린더 배너]가 실데이터 0건으로 비어 있음 → 수요 발굴이 그 배너에 실탄을 채움(신규 삽질 아님, 기존 자산 활성화).
> **원칙 정합**: 네이버 공식 API 2개만 사용(무료·약관 클린·PII 0) → 서버 최소·솔로 운영. 맘카페·자동완성 *대량 스크래핑은 비채택*(약관 회색+유지보수 부담).

## 엔진 = 네이버 공식 API 2개 (상보)

| API | 주는 것 | 못 주는 것 | 역할 |
|-----|--------|-----------|------|
| 검색광고 **키워드도구** (`/keywordstool`) | 연관키워드 + **절대 월검색량**(PC/모바일) + 경쟁도·CPC | 누가 검색하는지 | 발굴 + 규모 |
| 데이터랩 **검색어 트렌드** | **연령·성별·기기·시계열**(지수) | 절대량 | 시즌성·페르소나 검증 |

→ **반드시 조인**: "30대 여성이 이 시즌 급등시키는 키워드(데이터랩) × 그 절대 규모(키워드도구)" = 핵심 부모 페르소나의 진짜 수요.

## 시즌 캘린더 — 가설 (Step 2에서 데이터로 검증·확정)

🔴 = 진통제 윈도우(신청·마감·예매 = 시점성 강함, 발행은 N주 *선행* 필요)

| 월 | 핵심 수요(가설) | 비고 |
|----|----------------|------|
| **1월** | 🔴입학 준비·취학통지서·예비소집 / 겨울방학 특강 / 설 나들이 | 입학 D-60 |
| **2월** | 🔴늘봄·돌봄교실 신청 마감 / 어린이집·유치원 새학기 준비물 / 졸업·입학 | **진통제 피크** |
| **3월** | 새학기 적응(분리불안 급등 가설) / 학기초 학원 등록 | 정서 키워드 스파이크 |
| **4월** | 봄 체험·나들이 / 🔴어린이날 공연·전시 예매 시작 | 예매 선행 |
| **5월** | 🔴어린이날·가정의달(공연·전시·나들이 피크) | 연중 최대 이벤트 |
| **6월** | 여름 대비·물놀이 / 🔴여름방학 캠프·특강 신청 시작 | 1학기 마무리 |
| **7~8월** | 여름방학 캠프 / 키즈풀빌라·물놀이 / 아이와 여행 | 휴가 시즌 |
| **9월** | 2학기 적응 / 가을 체험학습 / 추석 나들이 | |
| **10월** | 가을 나들이·축제 / 핼러윈(키즈) | |
| **11월** | 겨울 대비 / 🔴겨울 공연 예매(호두까기인형 등) / 영어유치원 전형 | |
| **12월** | 🔴크리스마스(공연·선물·그림책 피크) / 겨울방학 특강 / 🔴취학통지서·예비소집 안내 / 전집 구매 | 입·출력 동시 피크 |

핵심 통찰(가설): **2월·12월이 진통제 쌍봉**(입학·신청·마감 밀집). 발행 캘린더는 이 윈도우를 *몇 주 앞서* 비워둬야 함.

## seed 키워드 세트
- 머신리더블: [`packages/db/scripts/demand/seed-keywords.ts`](../packages/db/scripts/demand/seed-keywords.ts) — 8카테고리 ~50개, 부모 실제 검색어 기준. 각 seed에 `hypothesisMonths`·`painkiller` 가설 태깅(위 캘린더와 1:1).
- Step 2 진단 스크립트가 이 배열을 그대로 API에 투입.

## 4단계 플랜 (현재 = Step 0 완료)

- **[x] Step 0 — seed 세트 + 시즌 캘린더 가설** (이 문서 + seed 파일). 키 불필요.
- **[ ] Step 1 — API 키 발급** *(당신만 가능, 유일 블로커)*
  - 네이버 검색광고: `NAVER_AD_CUSTOMER_ID` · `NAVER_AD_API_KEY` · `NAVER_AD_SECRET_KEY` (광고주센터 → 도구 → API 사용 관리)
  - 네이버 Developers 데이터랩: `NAVER_DATALAB_CLIENT_ID` · `NAVER_DATALAB_CLIENT_SECRET` (검색어 트렌드 API 신청)
  - → `.env.example`에 키 이름 문서화 후 `.env`에 실값.
- **[ ] Step 2 — 1회성 진단 스크립트** (키 나오면). 산출물: ⓐ 시즌 캘린더 *확정판* ⓑ 우리 큐레이션 vs 실수요 *정렬 격차*.
- **[ ] Step 3 — 시즌 캘린더 → 키데이트 배너·발행 우선순위 배선**. 첫 사용자 가치(시점성).
- **[ ] Step 4 — 주간 수요 잡 + 리서치 에이전트 주입** (Step 2가 가치 증명 후에만). 전담 "트렌드 스카우트" 에이전트 승격은 트리거 충족 시 별도 판단.

## 전담 에이전트?
지금은 **아니오**. 키워드/검색량은 결정론적 API+크런칭이라 LLM 불필요. LLM 가치는 *해석*(키워드 군집 → 8카테고리/5타입 매핑 + "왜 공명") 뿐 → 기존 리서치(스카우트) 에이전트 프롬프트에 "이번 주 수요 신호" 입력으로 주입. 독립 직군 승격은 (a)코퍼스 비대화 (b)별도 주기(주간) (c)큐레이션 적중률 측정 향상이 보일 때.
