# 페런트웹 데이터·프라이버시 원칙 (PRIVACY)

> **역할**: 아동 민감정보를 다루는 제품의 *데이터 아키텍처 + 프라이버시* 가드레일. 출시 직전 보안 게이트(`/security-review`)가 이 문서를 집행. 공통 정책이므로 [CLAUDE.md](../CLAUDE.md)와 한 몸(요약은 CLAUDE.md에도 흡수).
>
> **상태 기준일**: 2026-05-21 (씨앗). 트리거 = 솔로+Claude Code 운영 + "보안이 가장 크리티컬" 원칙.

## 0. 두 대원칙 \[확정 2026-05-21]

1. **솔로+Claude Code 감당 범주 안에서만** 설계. 순차 진행(병렬 X). 전문성은 채용 아닌 on-demand 패스.
2. **서버는 *서비스 운영 최소 정보*만. 나머지(특히 아동 PII)는 클라이언트(기기)에.** 두 원칙에 버든되는 기능은 **과감히 제거/연기.**

## 1. 핵심 통찰: 개인화 = 순수 함수 → 클라에서 연산

청사진 개인화 = (아이 생일·지역 = 민감) × (마일스톤 규칙 = 공개·정적). 후자는 공개 데이터라 번들 가능, 계산은 **클라이언트에서**. → **서버가 아이 PII를 알 필요 없음.**

## 2. 데이터 배치 \[확정]

| 데이터 | 위치 | 비고 |
|---|---|---|
| 아이 프로필(생일·지역·준비도·진단) | **클라이언트(기기)만** | 서버 미보관 |
| 마일스톤 규칙·발달 이정표 | 앱 번들/서버 | 공개·정적, 비민감 |
| 큐레이션 콘텐츠(브리핑·이벤트) | 서버 | 공개, 모두 동일 |
| 개인화 계산(생일+규칙→타임라인) | **클라이언트** | 로컬 연산 |
| 푸시 토큰·분석 | 서버(익명, 신원 미연결) | PII 아님 |
| (선택) 백업/싱크 | 안 함 or **E2E 암호화 블롭** | 서버가 못 읽음 |

**서버 환원 불가능 최소** = 공개 콘텐츠 + 정적 규칙 + 익명 토큰/분석. **아이 PII = 0.**

## 3. 기능별 원칙 정합 해법 \[확정]

- **마일스톤 D-day 알림 = 로컬 알림**(클라가 생일+규칙으로 기기서 스케줄). 서버 PII 0.
- **그 동네 새 이벤트 = 토픽 푸시**(지역 토픽 구독 + 클라 필터). 1:1 PII 0.
- **소비자 계정/로그인 = MVP 불필요**(기기 로컬 신원). 로그인 0 = 크리덴셜 0. 기존 Supabase Auth는 *운영자 어드민*용만.
- **싱크/백업** = MVP는 사용자 주도 내보내기/가져오기. E2E 암호화 싱크는 수요 시.

## 4. 과감히 제거/연기 (원칙 위배분) \[확정]

- ❌ 소비자 계정·로그인 → 기기 로컬 신원
- ❌ 서버측 아이 프로필 → 절대 안 둠
- ❌ 크로스기기 싱크 → MVP 수동 내보내기, E2E 나중
- ❌ 서버 스케줄 1:1 푸시 → 로컬 알림 + 토픽 푸시
- ❌ 커뮤니티/소셜(이미 V2) → 유지 제외(PII·모더레이션 무거움)

## 5. 브랜드 정합

"**네 아이 데이터는 네 폰에만, 우리 서버엔 없어**" = 안티-맘카페·신뢰(B 레이어)와 한 몸. 제약을 *차별점·마케팅 자산*으로 전환.

## 6. 남은 의무 (최소지만 0 아님) \[검토중]

- 명확한 개인정보처리방침 + 온디바이스 동의 UX(만 14세 미만 = 법정대리인). 서버 PII가 거의 없어 노출은 작지만 *방침·동의 절차*는 필요.
- 익명 분석도 *재식별 불가* 보장(생일·지역 조합이 식별자 되지 않게 coarse 처리).
- **출시 전 1회성 법무·보안 검토**(채용 아닌 on-demand) 권장.
- 로컬 전용의 최대 리스크 = **데이터 소실(폰 교체)** → 내보내기/백업 UX를 *반드시* 잘 만들 것(수년치 청사진 투자 보호).
