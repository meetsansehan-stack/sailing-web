## 프로젝트 목적
매일 자동으로 수집·큐레이션한 육아 관련 뉴스를
웹사이트에서 한눈에 볼 수 있는 큐레이션 플랫폼.

## MVP 핵심 잡
**캐치업 + 둘러보기** — 양육자가 5~10분 안에 "내가 모르고 지나간 정책/이슈가 없는지" 확인(캐치업). 더 보고 싶은 유저는 그날 발행분 전체를 둘러봄(둘러보기). **연령·지역 개인화는 V1 핵심**(2026-05-21 승격, 기존 "개인화 V2"를 덮어씀) — 단 *하드 필터 아닌 정렬·강조*, 클라이언트(localStorage/기기) 기반 익명 처리(서버 아동 PII 0 — 부모 계정 소셜 로그인은 별개, 데이터·프라이버시 §참조). 상세 STRATEGY §4·PRIVACY.

## 타겟 독자
- **만 3~9세 (어린이집·유치원 ~ 초등 저학년) 자녀를 둔 부모 (주 독자)**. **2026-06-12 재정의** (기존 "0~8세 [확정]"을 덮어씀): 만 0~2세 영아는 *완전 보육 단계*라 정보 니즈(수유·수면·이유식)가 질적으로 달라 제외. **만 3세(첫 기관 생활)**부터가 어린이집·유치원·취학 준비·교육 고민이 시작되는 구간 = 우리 8개 카테고리·진통제(취학 전후)·브랜드("좌표→수평선")와 정합. 니치 좁힘 = 공명↑ (PMF 진단 "병목=공명>TAM"과 정합).
- **V1 = 부모 페르소나에 집중** (2026-05-21). 교사·교육자 페르소나는 V2/재검토. (앱 청사진은 와우 페르소나를 만 5~8세 부모로 더 좁힘 — APP_SPEC §1.0. 웹 브리핑 타겟은 만 3~9.)

## 커버리지 카테고리 (8개, 라벨 + 필터로만 사용)
카테고리 수는 가변(향후 데이터 기반 운영, 최소 5개). 코드 단일 소스: `packages/shared/src/article.ts`.
- **교육 정책** `policy` (정책·제도·공교육 / 늘봄·돌봄·누리과정)
- **양육·발달** `parenting` (양육 노하우·발달·정서·전문가 콘텐츠)
- **학원·EdTech** `academy` (사교육 / 영어유치원·학원 트렌드·에듀테크)
- **놀이·체험** `play` (놀이터·키즈카페·장난감·체험형 활동)
- **공연·전시** `shows` (어린이 공연·뮤지컬·전시·박물관 행사)
- **도서그림책** `books` (그림책·아동도서·도서관 프로그램·작가)
- **시장·트렌드** `market` (육아용품 시장·스타트업·소비 동향)
- **기타** `others` (7개 분류에 안 맞지만 중요한 이슈를 버리지 않는 안전망 — 정말 분류 불가일 때만)

Category 코드값은 영어 slug. UI 라벨은 `CATEGORY_LABEL` 매핑으로 분리 (URL 가독성·SEO).

V1에서 카테고리는 **카드 라벨 + 쿼리스트링 필터**로만 사용. 별도 카테고리 페이지는 만들지 않음.

## 콘텐츠 타입 (5개)
다양성 보장을 위한 직교 차원. Article의 `contentType` 필드.

- **Policy** 정책·제도·공공 발표
- **Event** 시기 임박 이벤트 (공연·전시·축제) — 발행일 대신 *이벤트 시작일* 기준 미래 4주
- **Market** 시장·산업·소비 동향
- **Insight** 트렌드·분석·연구·조사
- **Guide** 실용 팁·체크리스트·액션 가이드

### 일별 권장 분포 (소프트 가이드)
- 5개 타입을 고루 — 특정 타입 쏠림 지양 (예: Policy·Event·Insight 각 2~4, Market·Guide 1~2). 발행량(8~20)에 비례 조정
- 주간으로 5개 타입 모두 1회 이상 등장 권장
- 하드 쿼터 X — 그날 약한 타입은 비워두고 이슈성·신뢰도가 우선

## 발행 정책
- **양**: 매일 8~20개 (큐레이션 우선, 카테고리 균형보다 그날 가장 중요한 것). 상수는 `packages/shared` config(`PUBLISH_MIN`/`PUBLISH_MAX`) 단일 관리 — soft, 재조정 가능
- **채널**: 웹사이트 매일 업데이트만 (이메일 뉴스레터는 V2)
- **시기성 윈도우**: 1주 (Event는 시작일 기준 미래 4주)
- **큐레이션 강도**: 매일 후보 ~40개 → 8~20개 (선별배율 ×2, 압축률 ≈50%). research 후보 미달은 실패 아님(품질 > 수량)

## IA / 라우트
- `/` → 오늘의 이슈 (8~20개 카드 메인)
- `/issues` → 아카이브 (과거 이슈 그리드, 최신순)
- `/issues/[YYYY-MM-DD]` → 특정 일자 이슈
- `/articles/[id]` → 기사 상세

## 데이터 모델
### Issue 엔티티
```ts
type Issue = {
  date: string;          // YYYY-MM-DD (식별자)
  title?: string;        // 그날의 한 줄 후킹 (hooking 에이전트)
  summary?: string;      // 그날의 짧은 요약 (hooking 에이전트)
  articleIds: string[];  // 5~7개
};
```

### Article 필수·선택 필드
- 필수: `id`, `title`, `summary`, `body`, `url`, `category`, `source`, `publishedAt`, `credibilityScore`, `contentType`, `issueDate`
- Event 타입 한정: `eventStartDate`
- 미디어 형식: `mediaType?` (`'text' | 'video' | 'podcast' | 'book'`, 미지정 시 `text` 가정)
- video·podcast 한정: `durationMin?` (분 단위)
- 선택: `tags?` (V2 필터 누적용)

## 본문 작성 스타일 (필수) — 2026-06-04 상세페이지 리뉴얼 반영

> **분기**: **홈/아카이브 카드 = 제목만**(디스크립션·본문 미표시) / **상세페이지 본문 = 세일링 산문 + 한눈에 불릿 + 특수 섹션**. (기존 "산문체 X, 전부 불릿"을 덮어씀 — 상세는 읽을거리. 상세 SPEC §10, `agents/writer/prompt.md`, `apps/web/src/components/ArticleBody.tsx`.)
> **공통 종결**: **친근한 존대**("~해요/하세요/좋아요") — 혜택·사실 프레이밍, 공포·재촉 아님. 격식체·명사형 금지. (2026-05-29 CPO 확정, docs/brand-voice.md §0·§2·§5)

상세페이지가 **헤딩 이름으로 섹션을 다르게 렌더**한다(헤딩 이름·순서·형식 = 렌더 계약, 어기면 깨짐):
- **한눈에 보는 핵심** (비Event 첫 섹션) = **3불릿** TL;DR → 상단 요약 박스(스캔). 본문에선 숨김.
- **서술 섹션**(시행 개요·달라지는 점·분석 핵심·핵심 콘텐츠·핵심 트렌드 등) = **세일링 산문**(불릿 아님, 읽기).
- **지표 섹션**(시장 지표·시장 현황·근거·데이터·지원 단가) = **"라벨: 수치" 불릿** → stat 표(숫자 강조).
- **신청 단계/방법** = **"1단계:" 번호 불릿** → 번호 배지.
- **양육자 체크·시사점·활용 팁·방문 팁** = so-what 2~4문장 → 🧭 콜아웃.
- **Event**: 행사 개요(경험 요약 불릿 → 상단 박스 흡수) / 핵심 콘텐츠(산문+이미지영역) / 방문 팁(중복없는 1~2줄). "한눈에" 섹션 안 씀.

## 출처 풀 (타입별 매핑)
출처는 **세 카테고리**로 묶여 화이트리스트 운영. 신뢰도 가시화·자극성 배제 정책은 모든 카테고리에 동일 적용.

### 카테고리 A — 공영방송·공공기관
- EBS 키즈·EBS 부모, KBS 다큐 키즈, 보건복지부·교육부 공식 영상 채널
- 신뢰도 0.85~0.95, 광고성 ZERO

### 카테고리 B — 검증된 전문가 채널 (의료·심리·교육)
- 오은영(소아청소년정신과), 김붕년(서울대 정신건강의학), 손석한, 조선미(아주대), 임영주(부모교육)
- 본인 운영 채널·언론 칼럼·도서 위주 / 협찬·예능 출연분 제외
- 신뢰도 0.80~0.90

### 카테고리 C — 도서 평론·큐레이션
- 한미화, 김지영(어린이책 평론가), 어린이도서연구회, 출판사 공식 큐레이션
- 도서그림책 카테고리 강화 / 신뢰도 0.75~0.85

### "둘 다 품되 선 긋기" 톤 가드레일 (웹·앱 공통)
사교육·입학·또래비교 등 부모가 진짜 궁금한 *현실 정보는 회피하지 않되*, 프레이밍은 **차분한 이해·발달 중심**(경쟁·불안·과시 배제). 특히 `학원·EdTech`·Insight·Market에 적용.
- "또래 대비"는 순위·등수 아닌 **참고 분포**로만 ("이 시기쯤 보통 ~함", 빠름/늦음에 정답 없음).
- **편집(큐레이션) 콘텐츠에서 사교육은 통계·데이터 비주얼화**(참여율·트렌드·분포 등 객관 지표)로 다룸. 개별 학원 나열·추천 X. "안 하면 뒤처짐/큰일" 류 공포·과시 카피 금지.
- **미래 교육·입시 흐름은 품되 현재 입시 공략은 긋는다 (2026-06-12, 시간축 구분)**: 타겟 독자(만 3~9 부모)지만 *주제*는 "우리 아이가 *그때* 겪을 교육·입시 제도 변화"(대입 개편 방향·고교학점제·신설/폐지)까지 품음 — 어린 부모도 미래 좌표("먼 수평선")로 직접 관심(검증된 유저 시그널). ❌ 단 *현재* 고등·대학생 대상 입시 공략·실행 테크닉(수능 전략 등)은 배제(우리 아이가 지금 겪는 게 아님=다른 고객·콘텐츠 희석). 미래 제도 변화는 반드시 **정보·준비 톤**(차분), "지금부터 안 하면 늦는다" 공포·재촉 금지(아래 불안vs공포 가드레일).
- **학원 정보·매칭은 직접 제공하지 않음.** 해당 서비스와 **제휴한 유료 광고 모델(BM)**로 분리하고, 편집 콘텐츠와 **명확히 구분 표기**. (시점·형태 V2, STRATEGY 참조.)
- → 포지셔닝 = "중립 데이터 매체". 학원 리스트의 책임·이해상충은 제휴 광고 레이어로 격리.
- **죄책감 면죄부 (Sailing 보이스)**: 양육자가 위축될 지점엔 "이미 늦음" 대신 **"지금·오늘부터"** 미래지향 한 줄. 과거형 "못 해줬다" 프레이밍 금지.
- **불안 공감 ✅ vs 공포 자극 ❌**: 양육 일상의 감정엔 거울처럼 공감하되, 불안의 대상이 'AI·미래·위협'이면 공포 자극으로 배제. 중요도는 *사실·규모·시점*으로 전달. 상세 [docs/brand-voice.md](docs/brand-voice.md).

## 데이터·프라이버시 원칙 (웹·앱 공통) — 상세 [docs/PRIVACY.md](docs/PRIVACY.md)
보안이 가장 크리티컬. **서버는 서비스 운영 최소 정보만, 나머지(특히 아동 PII)는 클라이언트(기기).** 두 원칙(솔로+Claude Code 감당 범주 / 서버 최소)에 버든되는 기능은 **과감히 제거·연기.**
- **로컬-퍼스트**: 아이 프로필(생일·지역·발달·진단)은 *기기에만*. 개인화는 (생일·지역)×(공개 마일스톤 규칙)의 *클라이언트 연산* → 서버가 아이 PII 알 필요 없음.
- 서버 최소 = 공개 콘텐츠 + 정적 규칙 + 익명 토큰/분석. **아이 PII = 0.**
- 마일스톤 알림 = 로컬 알림 / 동네 이벤트 = 토픽 푸시 / 싱크 = 수동 내보내기, E2E는 나중.
- **부모 계정 (2026-05-29 수정 — 기존 "소비자 계정 불필요"를 덮어씀)**: 뉴닉식 **소셜 로그인(카카오·구글) = 선택·점진형(progressive)** 도입. 익명 읽기 먼저(획득 안 막음) → 구독·저장·싱크 시점에 로그인 유도(리텐션·이메일 V2 토대). **계정 ≠ 아동 PII**: 서버엔 부모 신원(이메일·소셜ID·동의 플래그·거친 선호)만, **아동 PII(생일·발달·진단)·연령/지역 개인화 입력은 여전히 로컬-퍼스트**(동기화는 선택, E2E 나중). Supabase Auth(소비자+운영자), 비번 직접 저장 X. 운영자 어드민은 `/admin` 분리.
- 남은 의무: 개인정보처리방침·동의 UX(만14세미만 법정대리인), 익명 재식별 불가, 출시 전 1회 법무·보안 검토, 데이터 소실 방지 백업 UX.

## 기술 스택
- Frontend: Next.js 14, TypeScript, Tailwind CSS
- Backend: Node.js, Hono
- DB: PostgreSQL, Prisma, Redis (선택)
- AI: Claude Agent SDK. 에이전트별 모델 = `packages/agents-core/src/models.ts` 단일 소스 (성능·효율 기준: 큐레이션·집필·에디터·후킹=Opus 4.8 / 리서치·QA=Sonnet 4.6, env로 개별 오버라이드). 실호출 검증은 Gate①(결제/카드) 이후.
- Deploy: Vercel + Railway

## 에이전트 파이프라인
① 리서치 → ② 큐레이션 → ③ 집필(writer) → ④ 에디터 → ⑤ 후킹
(직군: 스카우트 / 데스크 / 기자 / 카피교열 / 소셜PD. 상세 동작은 docs/SPEC.md)

## 발행 스케줄 (KST)
- 00:00 리서치 에이전트 (뉴스 수집, 후보 ~40개 best-effort)
- 01:00 큐레이션 에이전트 (8~20개 선별 + 콘텐츠 타입 확정 + Article draft 적재)
- 02:00 집필(writer) 에이전트 (기사 건당 카드뉴스 본문 — WebFetch로 원문 읽고 집필)
- 02:30 에디터 에이전트 (순서·테마·교열)
- 03:00 후킹 에이전트 (Issue 메타·일자별 후킹)
- 웹사이트: 실시간 업데이트

## 코딩 규칙
- 모든 파일은 TypeScript strict mode
- 에이전트 프롬프트는 agents/{name}/prompt.md에서 관리
- 에이전트 입출력은 agents/{name}/schema.ts에 Zod 스키마로 정의
- 환경변수는 .env.example에 항상 키 이름 문서화
- 컴포넌트는 150줄 이하로 유지
- 기사 전문은 DB에 저장하지 않음 (URL + 자체 요약·본문만)
